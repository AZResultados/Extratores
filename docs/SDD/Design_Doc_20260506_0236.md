## 0. Princípios de Arquitetura

### VBA é camada de conveniência temporária

O Excel/VBA é o backend de persistência do MVP por conveniência operacional — não por escolha arquitetural definitiva. O path de evolução planejado é:

```
MVP (agora)          → v1.0 (pós-validação)        → Produção
Excel/VBA (Windows)    Docker + Linux + banco DB      Container orquestrado
```

**Consequência direta para todo desenvolvimento:**

> Toda lógica de negócio reside exclusivamente em Python.  
> O VBA faz apenas três coisas: disparar o script Python, capturar stdout, gravar na planilha.

| Fica em Python | Nunca entra no VBA |
|---|---|
| Extração e parsing de PDF | Regras de validação de lançamentos |
| Descriptografia in-memory | Detecção de emissor |
| Montagem do envelope JSON | Lógica de parcela ou data |
| Validação de argumentos CLI | Qualquer cálculo financeiro |
| Roteamento entre extratores | Tratamento de erros de negócio |

**Por quê isso importa:** quando o backend migrar de Excel para banco de dados em Docker/Linux, os extratores Python não precisam ser reescritos — apenas o consumidor do stdout muda (de VBA para um conector de banco). Qualquer lógica colocada no VBA hoje vira retrabalho garantido na migração.

### Núcleo portável vs camada de conveniência

```
┌─────────────────────────────────────────┐
│  NÚCLEO PORTÁVEL (Python)               │
│  extrator.py  (entry point único)       │
│  pdf_router.py                          │
│  cartao_mercadopago.py                  │
│  cartao_santander.py                    │
│  cartao_samsung.py                      │
│  cartao_itau_personnalite.py            │
│  extrator_nubank_rdb.py                 │
│  pdf_decrypt.py                         │
│  logger.py                              │
│  → stdin/stdout/stderr                  │
│  → Path() para caminhos (cross-platform)│
└─────────────────────────────────────────┘
         ↕ JSON via stdout
┌─────────────────────────────────────────┐
│  CAMADA DE CONVENIÊNCIA (substituível)  │
│  MVP:       Excel VBA (Windows)         │
│  v1.0+:     conector DB (Docker/Linux)  │
└─────────────────────────────────────────┘
```

## 1. Arquitetura de Isolamento e Lote

Input Guard: Entrada obrigatória via diretório: input/[NOME_DO_CLIENTE]/.

Validação Cruzada (não-fatal): O Python deve validar se `args.cliente` aparece em qualquer componente do caminho de `input_path` (não apenas `path.parent.name` — permite subpastas mensais dentro da pasta do cliente).
Em caso de divergência, o processamento continua normalmente, mas um aviso é incluído no envelope JSON de saída como campo "avisos": ["AVISO: 'cliente' nao encontrado no caminho '...'. Verifique isolamento de dados."]
O VBA deve exibir esse aviso via MsgBox ao operador ao final do processo bem-sucedido.

Mecânica: O script processa todos os *.pdf da subpasta e injeta o identificador --cliente em cada linha.

## 2. Motor de I/O e Segurança (BR-06)

Memória Pura: Proibida a criação de arquivos temporários (PDF ou JSON) em disco.

Descriptografia centralizada: A lógica de abertura de PDF protegido por senha é implementada exclusivamente em `src/pdf_decrypt.py` — módulo compartilhado importado por todos os extratores. Nenhum extrator contém lógica própria de descriptografia.

Interface do módulo:
  `descriptografar(pdf_path: Path, password: str) → BytesIO | Path`
  - password vazio → retorna Path original sem modificação (PDF sem senha).
  - password preenchido → abre com pikepdf, salva em io.BytesIO, retorna buffer in-memory.
  - Falha (senha incorreta, arquivo corrompido) → sys.exit(1) + mensagem em stderr.

  `tentar_descriptografar(pdf_path: Path, password: str) → BytesIO | Path`
  - Mesmo comportamento, mas levanta Exception em vez de sys.exit — uso exclusivo do router.

Roteamento de emissor: `src/pdf_router.py` detecta o emissor pelo conteúdo textual do PDF (fingerprint), nunca pela senha que o abriu. Tenta abrir sem senha; se protegido, cicla pelas senhas cadastradas para o cliente no banco SQLite.

Credenciais: armazenadas em SQLite local (`~/.extratores/dados.db`), schema `(cliente, senha)`. Entregues ao Python via stdin — nunca em arg CLI nem em planilha.

Transporte: O JSON resultante viaja exclusivamente via stdout. Mensagens de erro fatais viajam via stderr.

## 2b. Logging Estruturado

Módulo: `src/logger.py`  
Arquivo de log: `~/.extratores/extrator.log`  
Formato: `%(asctime)s | %(levelname)-8s | %(name)s | %(message)s`  
Rotação: 5 MB, 3 backups (RotatingFileHandler)  
Nível padrão: INFO (override via env `EXTRATORES_LOG_LEVEL`)

Hierarquia de loggers:
- `extratores.extrator` — início/fim de lote, PDF processado
- `extratores.router`   — emissor detectado, avisos de senha
- `extratores.santander` / `extratores.mercadopago` / `extratores.nubank_rdb` / `extratores.itau_personnalite` / `extratores.samsung` — extração por arquivo
- `extratores.db`       — operações CRUD no banco

Restrições de segurança:
- **Nunca logar**: senhas, CPFs, números de cartão completos
- `final_cartao` (4 dígitos) pode aparecer apenas em nível DEBUG
- Falha no logging não propaga exceção — fallback para stderr

## 3. Protocolo de Integração VBA ↔ Python (BR-03)

Para evitar deadlocks de buffer e garantir o Fail-Fast, o fluxo no VBA deve seguir esta ordem estrita:

Execução: Disparar o script via WScript.Shell.Exec.

Captura de Streams:
  saida_json = Trim(oExec.StdOut.ReadAll)
  erro_texto = oExec.StdErr.ReadAll

Sincronização: A leitura completa dos streams (ReadAll) garante o término do processo. Não utilizar .Status = 0 para espera.

Avaliação de Saída:

  Se .ExitCode <> 0: Exibir erro_texto via MsgBox, descartar saida_json e encerrar sem alterar a planilha.

  Se .ExitCode = 0:
    1. Fazer o parse do JSON via `ParseExtratorJSON()` — parser VBA puro, sem dependências COM.
    2. Se o campo "avisos" estiver presente e não vazio, exibir os avisos via MsgBox.
    3. Prosseguir com a gravação atômica na planilha.

> Nota: este protocolo é específico do backend VBA/Excel (MVP). Na migração Docker/Linux, o consumidor do stdout será um conector de banco — o protocolo Python-side (stdout JSON + exit codes) permanece idêntico.

## 4. Estrutura do Envelope JSON

```json
{
  "id_lote": "SA-20260429-032100",
  "data_processamento": "2026-04-29T03:21:00",
  "emissor": "santander|mercadopago|nubank_rdb|itau_personnalite|multi",
  "cliente": "NOME-DO-CLIENTE",
  "avisos": [],
  "lancamentos": [
    {
      "cliente":            "NOME-DO-CLIENTE",
      "id_lote":            "SA-20260429-032100",
      "arquivo":            "fatura_04_2026.pdf",
      "titular":            "NOME TITULAR",
      "final_cartao":       "1234",
      "tipo":               "Compra parcelada",
      "data_compra":        "15/02/2026",
      "descricao":          "SUPERMERCADO XYZ",
      "parcela_num":        2,
      "qtde_parcelas":      6,
      "vencimento":         "10/05/2026",
      "descricao_adaptada": "SUPERMERCADO XYZ parc 2/6 15/02/2026",
      "valor":              -289.42
    }
  ]
}
```

Regras do envelope:
- `id_lote` formato `{EMISSOR}-{YYYYMMDD}-{HHMMSS}` — NÃO usar UUID
- `emissor` = `"multi"` quando a pasta contém PDFs de mais de um emissor
- `emissor` aceita também `nubank_rdb` (extrato de aplicação/caixinhas PJ Nubank) e `itau_personnalite`
- `avisos` sempre presente (lista vazia = execução limpa)
- `arquivo` = `Path.name` apenas — sem caminho absoluto (PII)
- `data_compra` = null quando impossível inferir
- `parcela_num` e `qtde_parcelas` = 0 para lançamentos não parcelados

> O envelope JSON é o contrato de interface entre o núcleo Python e qualquer backend. Não alterar campos existentes sem versionar o schema.

## 7. Extrator Nubank RDB

**Arquivo:** `src/extrator_nubank_rdb.py`  
**Emissor:** `nubank_rdb`  
**PDF:** sem senha (não requer pikepdf)

### 7.1 Campos fixos em todos os lançamentos

| Campo               | Valor                                                  |
|---------------------|--------------------------------------------------------|
| `titular`           | Extraído do PDF (razão social da empresa beneficiária) |
| `final_cartao`      | `null`                                                 |
| `parcela_num`       | `0`                                                    |
| `qtde_parcelas`     | `0`                                                    |
| `data_compra`       | Igual à data da linha                                  |
| `vencimento`        | Igual à data da linha                                  |
| `descricao_adaptada`| Igual à `descricao`                                    |

### 7.2 Geração de lançamentos por linha do PDF

| Movimentação              | Lançamentos gerados        | `tipo`  | Valor base         |
|---------------------------|----------------------------|---------|--------------------|
| Compra por aplicação      | 1× "Aplicação RDB"         | Entrada | Valor Bruto (+)    |
| Resgate                   | 1× "Resgate RDB"           | Saída   | Saldo Líquido (−)  |
| Resgate (IR > 0)          | + 1× "IR s/ Resgate RDB"   | Saída   | IR (−)             |
| Resgate (IOF > 0)         | + 1× "IOF s/ Resgate RDB"  | Saída   | IOF (−)            |
| Rendimento até essa data  | 1× "Rendimento RDB"        | Entrada | Rendimento (+)     |

> "Resgate RDB" usa o **Saldo Líquido** (valor efetivamente recebido pela empresa). IR e IOF são lançamentos separados. A soma dos três reconstitui o Valor Bruto: `saldo_líquido + IR + IOF = valor_bruto`.

### 7.3 Estratégia de parsing

O PDF Nubank RDB apresenta movimentações multilinhas: a descrição ("Compra por" / "Rendimento até") aparece em linha separada, antes ou após a linha com data e valores. O parser usa uma máquina de estado simples (`pendente`):

1. Ao detectar `"Compra por"` → sinaliza `pendente = "aplicacao"`
2. Ao detectar `"Rendimento até"` → sinaliza `pendente = "rendimento"`
3. Ao encontrar linha iniciada por `DD/MM/YYYY`:
   - Com `"Resgate"` inline → processa diretamente (linha completa)
   - Com `pendente = "aplicacao"` → captura 4 valores (bruto, IR, IOF, líquido); usa Valor Bruto
   - Com `pendente = "rendimento"` → captura 1 valor; gera lançamento de rendimento
4. Linhas de continuação (`"aplicação"`, `"essa data"`) são ignoradas automaticamente

Fingerprint no router: `"nubank_rdb": ["RDB Resgate Imediato", "Caixinhas PJ"]`

### 7.4 Validação

Duas camadas de validação, ambas com tolerância R$ 0,10:

**Per-row (sempre ativa):** para cada linha de resgate no PDF verifica `valor_bruto − IR − IOF = saldo_líquido`. Garante integridade do parsing independente de dados externos.

**Saldo do período (requer `--saldo-abertura`):**

```
saldo_abertura − Σ(resgates_bruto) + Σ(aplicações) + Σ(rendimentos) = saldo_final_pdf
```

Onde `Σ(resgates_bruto) = Σ(saldo_líquido) + Σ(IR) + Σ(IOF)` — reconstituído somando os três tipos de lançamento. Divergência > R$ 0,10 encerra com `sys.exit(1)` e mensagem detalhada em stderr.

### 7.5 Interface CLI

```
python extrator_nubank_rdb.py --input-dir <path> --cliente <nome> [--saldo-abertura <float>]
```

- `--password` aceito mas ignorado (PDF sem senha)
- `--saldo-abertura` opcional; habilita validação completa de saldo do período
- Saída: `sys.stdout.write(json.dumps(envelope, ensure_ascii=True))`
- `processar_arquivo(pdf_path, source, saldo_abertura=None)` disponível para integração futura em `extrator.py`

## 8. Extrator Itaú Personnalitê

**Arquivo:** `src/cartao_itau_personnalite.py`  
**Emissor:** `itau_personnalite`  
**Escopo:** linha Personnalitê do Itaú Unibanco — qualquer bandeira (Mastercard, Visa, Elo) e categoria (Black, Platinum, Gold, etc.)  
**PDF:** protegido por senha — descriptografia via `pdf_decrypt.py` (senha no banco SQLite)

### 8.1 Decisões locked

| ID  | Decisão |
|-----|---------|
| I1  | `emissor = "itau_personnalite"` |
| I2  | `id_lote` prefix: `"ITP"` |
| I3  | `X_DIV = 355` — confirmado por medição; não herdar de outros extratores |
| I4  | Fingerprint: `["40044828", "ITAUUNIBANCOHOLDING"]` — confirmado no recon (TASK-IT-01); `"Personnalitê"` não existe no text layer (é logo/imagem) |
| I5  | Escopo: qualquer bandeira/categoria da linha Personnalitê |
| I6  | Descriptografia via `pdf_decrypt.py` — senha no banco SQLite, padrão idêntico ao Santander/Samsung |
| I7  | Múltiplos titulares: rastrear `titular_ativo` e `final_cartao_ativo` por bloco intermediário |
| I8  | Seção `"Compras parceladas - próximas faturas"` ignorada integralmente — para ambas as colunas ao ser detectada em qualquer uma |
| I9  | Ajustes negativos sem parcela → `tipo = "Ajuste"` → valor armazenado como positivo (crédito) |
| I10 | Validação: `"Total dos lançamentos atuais"` com tolerância R$ 0,10; ausência = WARNING, não abort |
| I11 | `x_tolerance=1` em `extract_words()` — necessário para preservar espaços nas descrições; detecção de padrões usa `''.join(tokens)` |

### 8.2 Layout de colunas

Medido com `extract_words()` na página 2 do PDF de teste:

```
Coluna ESQ:  x0 ≈ 151 (DATA) | x0 ≈ 178 (ESTABELECIMENTO) | x0 ≈ 225–234 (PARCELA) | x0 ≈ 319–323 (VALOR)
Coluna DIR:  x0 ≈ 367 (DATA) | x0 ≈ 394 (ESTABELECIMENTO) | x0 ≈ 530–556 (VALOR)

X_DIV = 355  ← gap real medido: 327 a 367
```

> Diferente do Samsung (330) e do Santander (250). Não reutilizar constante de outros extratores.

### 8.3 Extração de titular e cartão (página 1)

`extract_text()` retorna os campos da página 1 com espaços corretos. Regex aplicados sobre o texto extraído por `extract_text()`:

```python
RE_TITUL = re.compile(r"Titular\s+([A-Z][A-Z\s]+)")
RE_CART  = re.compile(r"Cart[aã]o\s+\d{4}\.XXXX\.XXXX\.(\d{4})")
RE_VENC  = re.compile(r"Vencimento\s+(\d{2}/\d{2}/\d{4})")   # sem ":" — boleto da pág. 1
```

`final_cartao`: últimos 4 dígitos do padrão `NNNN.XXXX.XXXX.NNNN`.

### 8.4 Estrutura multi-titular

A fatura Personnalitê agrupa lançamentos por portador com cabeçalhos e subtotais intermediários.

```
NOME PORTADOR (final NNNN)         ← cabeçalho de bloco: atualizar titular_ativo
  DATA  ESTABELECIMENTO  VALOR
  ...lançamentos deste bloco...
Lançamentos no cartão (final NNNN)  SUBTOTAL   ← linha de subtotal: ignorar
OUTRO PORTADOR (final NNNN)
  ...
Total dos lançamentos atuais        TOTAL GERAL  ← campo de validação
```

Regras de parsing:
- Ao detectar cabeçalho de bloco: atualizar `titular_ativo` e `final_cartao_ativo`
- Nomes de titulares extraídos com espaços via `' '.join(tokens[:idx_final])`
- Linha `"Lançamentos no cartão"`: pular
- Linha `"Total dos lançamentos atuais"` (texto unido: `"Totaldoslançamentosatuais"`): capturar para validação

> Este padrão multi-titular não é exclusivo do Itaú — o Samsung pode apresentar o mesmo comportamento quando o cliente possui cartões adicionais ativos.

### 8.5 Seções a processar / ignorar

| Seção | Ação |
|-------|------|
| `Lançamentos: compras e saques` | ✅ Processar |
| `Compras parceladas - próximas faturas` | ❌ Ignorar — para a coluna atual com `break` e bloqueia col direita |
| `Pagamento efetuado em DD/MM/YYYY` | ✅ Capturar como tipo `Pagamento` |
| Demais seções (Encargos, Limites, Simulações) | ❌ Ignorar |

### 8.6 Padrão de linha de lançamento

```
DATA  NOME_ESTABELECIMENTO[PARCELA]  VALOR
```

- `PARCELA`: token `NN/NN` como penúltimo token (detectado por RE_PARC)
- Lançamentos de ajuste: valor negativo sem parcela — ex: `HOSPITAL RUBEM BERTA -1.033,44`
- Valores negativos no PDF (`-0,02`) podem ser fragmentados por `x_tolerance=1` em `['-', '0,02']`; reagrupados em `extrair_segmento()` antes de chegar ao parser
- Linhas de categoria (ALIMENTAÇÃO, VESTUÁRIO, etc.) filtradas por ausência de data no primeiro token

### 8.7 Classificação de tipo e convenção de valor

| Condição | Tipo | Valor armazenado |
|----------|------|-----------------|
| `valor > 0` e `parcela_num > 0` | `"Compra parcelada"` | `-abs(valor)` |
| `valor > 0` e `parcela_num = 0` | `"Compra à vista"` | `-abs(valor)` |
| `valor < 0` e sem parcela | `"Ajuste"` | `+abs(valor)` (crédito) |
| Descrição contém "pagamento" | `"Pagamento"` | `+abs(valor)` |

Fórmula de validação: `total_pdf = -Σ(valor para tipo != Pagamento)` = compras − ajustes.

### 8.8 Validação

- **Campo:** `"Total dos lançamentos atuais"` (pode aparecer como `"LTotaldoslançamentosatuais"` por artefato de layout)
- **Regex:** `r"L?Totaldoslançamentosatuais\s+([\d.]+,\d{2})"`
- **Tolerância:** R$ 0,10
- **Ausência:** logar WARNING, retornar `None` — não abortar
